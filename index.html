§ 02 — Стратегические сдвиги

После OpenAI Workspace Agents

Запуск Workspace Agents в ноябре 2025 переопределил рынок: агент перестал быть чат-ботом и стал членом команды. Вот что меняется в нашем продукте.

Агент = член команды, а не чат
Persistent identity, профиль, «рабочее место», history, mentions @ии-юрист
Async long-running tasks
Очередь задач, push-уведомления, агент работает часы и дни
Multi-agent collaboration
Оркестратор, агент-агент протокол, «комитеты» (как у нас уже в Инвест-менеджере)
Shared workspace
Совместные пространства «человек + агенты», каналы по проектам, общий контекст
Connectors как первый класс
Маркетплейс к РФ-системам: 1С, SAP, Directum, Bitrix, ELMA, Диадок, КонсультантПлюс
Approval flows
Human-in-the-loop для action-агентов: согласование перед отправкой письма / проводки / договора
Governance & audit
DLP, RBAC до уровня документа, audit trail каждого действия, политики ИБ

EPIC-01 Unified Hub Shell — единая точка входа Now RICE 7.5

Обоснование

Сегодня каждый копилот — отдельный UI. На демо клиенты путаются: «это разные продукты или один?». Без единого шелла нельзя продавать как платформу — продаём как набор разрозненных решений с нулевым cross-sell.

User Stories

  • US-01.1Как сотрудник я открываю один URL и вижу все доступные мне копилоты (по правам), переключаюсь без повторного логина (SSO).
  • US-01.2Как админ я в одной консоли назначаю какие копилоты доступны каким группам AD/Keycloak.
  • US-01.3Как пользователь я ищу по всем своим чатам со всеми копилотами одной строкой.
  • US-01.4Как пользователь я закрепляю избранные копилоты на главном экране.
  • US-01.5Как сотрудник я вижу единую историю запросов с фильтром по копилоту/проекту/дате.

Acceptance criteria

SSO через Keycloak/AD FS/ЕСИА; единая навигация; глобальный поиск ≤2 сек; mobile-responsive.

EPIC-02 On-prem Reference Architecture & Deployment Toolkit Now RICE 5.4

Обоснование

On-premise — наш ключевой дифференциатор vs OpenAI/Anthropic и vs облачных GigaChat/YandexGPT для гос. Сейчас каждое внедрение — кастомный проект на 2–4 месяца. Цель: time-to-deploy с 90 до 14 дней.

User Stories

  • US-02.1DevOps клиента разворачивает Hub в Kubernetes одним helm-чартом под RHEL/Astra Linux/РЕД ОС.
  • US-02.2ИБ-офицер получает готовый пакет: модель угроз, регламент, инструкции пользователю.
  • US-02.3DevOps подключает встроенную/внешнюю Cotype или открытую LLM (Qwen, GigaChat on-prem) переключением конфига.
  • US-02.4Админ мониторит утилизацию GPU/RAM в Grafana (готовые дашборды).
  • US-02.5Клиент получает SLA-плейбук для air-gap-окружений.

EPIC-04 Agent Identity & Mentions — «агент как сотрудник» Now RICE 8.0

Обоснование

Главный сдвиг от Workspace Agents — агент имеет имя, профиль, аватар. Это не косметика: меняет восприятие с «инструмент» на «коллега» → растёт engagement и DAU. Дешёвая фича с большим эффектом.

User Stories

  • US-04.1Админ даёт агенту имя, аватар, описание роли.
  • US-04.2Сотрудник упоминает @юрист-аня в чате и втягивает агента в обсуждение.
  • US-04.3Сотрудник видит карточку агента: что умеет, к каким данным имеет доступ, кто owner.
  • US-04.4Админ создаёт несколько инстансов одного агента под разные департаменты.
EPIC-05 Async Tasks & Long-running Agents Now RICE 3.36

Обоснование

Workspace Agents продают именно long-running. Сегодня все наши копилоты — синхронные (≤60 сек). Это потолок ценности. Async открывает кейсы: «проверь 500 договоров за ночь», «собери досье на 30 кандидатов».

User Stories

  • US-05.1Пользователь ставит задачу «проверь 200 договоров в папке Q2» и закрывает окно — получает push, когда готово.
  • US-05.2Видит панель «Мои задачи» с прогрессом каждого агента.
  • US-05.3Может прервать долгую задачу или скорректировать на лету.
  • US-05.4Админ ставит квоты на длительность/стоимость задач по группам.
EPIC-06 Approval Flows & Human-in-the-loop Now RICE 5.3

Обоснование

Без approval-flows нельзя дать агенту action-tools (отправить письмо, провести проводку). Это блокер перехода от «ассистент» к «агент». Юр. отдел и ИБ требуют это первым же вопросом на пилоте.

User Stories

  • US-06.1Админ настраивает политику: «проводки >100 тыс ₽ требуют подтверждения главбуха».
  • US-06.2Менеджер получает в Telegram/почту запрос approve/reject с превью и обоснованием.
  • US-06.3Аудитор видит полный лог: что предложил агент, кто approved, когда выполнено.
  • US-06.4Пользователь может затребовать «explain» — почему агент предлагает именно это.
EPIC-07 Production-grade Eval & Quality Gate Now RICE 4.0

Обоснование

Заявленные 97% качества — маркетинг. В реальности нет регулярного eval по версиям. Каждое обновление модели = риск регрессий у клиентов. Без этого нельзя продавать SLA по качеству.

User Stories

  • US-07.1ML-инженер запускает золотой набор (≥500 кейсов на копилот) автоматически на каждом релизе.
  • US-07.2PM видит dashboard: precision/recall/halluc-rate по версиям.
  • US-07.3Клиент получает отчёт по качеству на своих данных раз в месяц.
  • US-07.4Можно прогнать A/B двух версий промпта на проде с 5% трафика.
EPIC-08 RAG 2.0 — гибридный поиск + structured + цитирование Now RICE 3.3

Обоснование

RAG — главный ROI-драйвер для всех копилотов. Сейчас vanilla vector-search → промахи на нумерованных пунктах ЛНА, табличных данных, доменных аббревиатурах. Юристам критично цитирование «пункт 4.2.1». Финансистам — drill-down до проводки.

User Stories

  • US-08.1Юрист-копилот отдаёт ответ с прямыми ссылками на пункты документа (anchor-level citation).
  • US-08.2Финансист может drill-down от ответа до проводки в 1С.
  • US-08.3Админ добавляет свой словарь синонимов/аббревиатур.
  • US-08.4RAG-движок учитывает структуру документа (заголовки, таблицы, нумерация).
  • US-08.5Пользователь видит confidence score и явный отказ «не нашёл» вместо галлюцинации.
EPIC-09 Workspace Channels & Multi-agent Collaboration Next RICE 1.6

Обоснование

Главный архитектурный шаг к Workspace Agents-парадигме. Каналы по проектам, где работают люди + несколько агентов одновременно. У нас уже есть прототип multi-agent в Инвест-менеджере (5 аналитиков) — обобщаем как платформенную фичу.

User Stories

  • US-09.1PM проекта создаёт канал «Сделка X» и приглашает @ии-юрист, @ии-финансист, @ии-аналитик.
  • US-09.2Агент видит контекст канала и переписку других агентов, может к ним обращаться.
  • US-09.3Пользователь видит, когда один агент делегирует задачу другому.
  • US-09.4Админ настраивает allowed agent-to-agent communication (DLP).
EPIC-10 Copilot Marketplace — внутренний и партнёрский Next RICE 1.4

Обоснование

Закрывает long-tail вертикалей. Мы не сделаем копилотов под нефтегаз/металлургию/транспорт сами — даём платформу партнёрам (СИ, отраслевые вендоры). Аналог GPT Store, но enterprise. Источник marketplace-revenue 30/70.

User Stories

  • US-10.1Партнёр публикует копилот в маркетплейс, проходя ревью качества и ИБ.
  • US-10.2Админ клиента видит каталог, фильтрует по индустрии, ставит в свой контур одной кнопкой.
  • US-10.3Партнёр получает аналитику использования и revenue-share отчёт.
  • US-10.4MWS модерирует копилоты, отзывает при инцидентах.
EPIC-11 Agent Memory & Personalization Next RICE 3.6

Обоснование

Сейчас агент забывает контекст между сессиями. Workspace Agents — с persistent memory. Без памяти агент = stateless tool, с памятью = коллега, который тебя знает. Boost для DAU/retention.

User Stories

  • US-11.1Пользователь говорит «помни, что я отвечаю за регион Урал» — агент применяет это во всех будущих ответах.
  • US-11.2Видит/редактирует/удаляет, что агент о нём помнит.
  • US-11.3Админ задаёт политику, что агент НЕ должен запоминать (PII, медданные).
  • US-11.4Агент хранит project-level memory отдельно от user-level.
EPIC-12 Voice & Multi-modal Input Next RICE 1.9

Обоснование

Топ-менеджмент enterprise редко печатает. Голос на мобиле + загрузка фото документа = 10x adoption у C-level и полевых сотрудников (склад, ритейл, стройка). MWS уже имеет голосовые модели.

User Stories

  • US-12.1Руководитель диктует запрос в мобильном приложении.
  • US-12.2Кладовщик фотографирует накладную → ИИ-бухгалтер распознаёт и заводит.
  • US-12.3Сотрудник загружает скан договора → ИИ-юрист анализирует.
EPIC-13 Governance Console — RBAC, DLP, Audit Next RICE 6.0

Обоснование

Без enterprise-grade governance не продадим в банки/гос/нефтегаз. Это не feature — это hygiene. ИБ-офицер на сделке за 10 минут зарезает продукт без RBAC до уровня документа, DLP-политик и полного audit-trail.

User Stories

  • US-13.1ИБ-офицер видит полный лог: кто, что спросил, что агент ответил, какие данные читал, какие действия выполнил.
  • US-13.2Админ задаёт DLP-политики: «не показывать данные класса К1 пользователям без допуска».
  • US-13.3Админ экспортирует audit-log в SIEM (KUMA, MaxPatrol).
  • US-13.4ИБ настраивает PII-маскирование на лету.
  • US-13.5Дашборд «топ-10 рискованных запросов за неделю».
EPIC-14 ИИ-аналитик 2.0 — Text-to-SQL + BI Next RICE 1.6

Обоснование

Аналитик упомянут в семействе, но реально не построен. Спрос огромный: каждый CEO хочет «спросить у данных». Барьер — точность Text-to-SQL на грязных корп-схемах ~60%. Нужна семантическая модель и накопить датасет.

User Stories

  • US-14.1Руководитель спрашивает «выручка по СЗФО за март» — получает таблицу + график.
  • US-14.2Аналитик подключает Greenplum/ClickHouse/PostgreSQL и описывает семантический слой через UI.
  • US-14.3Пользователь видит explain — какой SQL агент сгенерил, почему.
  • US-14.4Админ ограничивает агенту таблицы/строки по ролям.
EPIC-15 Notification & Proactive Agents Next RICE 3.27

Обоснование

Эволюция от reactive («я спросил — ответил») к proactive («сам заметил — предупредил»). Уже частично есть в Финансист.Radar. Обобщаем: любой копилот может настроить триггер-наблюдение.

§ 05 — Технические требования

LLM-стек и железо

Принцип — мульти-модельный роутер, не моно-ставка. Разные задачи получают разные модели по соотношению качество / стоимость / латентность.

5 tier'ов моделей

Tier 1 — Reasoning
70B+ · контекст ≥128K · TTFT ≤2 сек

Тяжёлые задачи: юридический анализ, инвест-комитет, multi-step agents. MMLU ≥80, RU-MMLU ≥75. Обязательно нативный function calling и structured output.

Cotype ProGigaChat 2 MAX on-premQwen 2.5 72BDeepSeek V3Llama 3.3 70BT-Pro
Tier 2 — Workhorse
7–32B · контекст 32K · ≥60 tok/sec

Повседневные ответы Core Copilot, RAG-ответы.

Tier 3 — Embeddings & Rerankers
Для RAG · мультиязычные · 1024+ dim

Гибридный поиск (dense + sparse BM25), кросс-энкодер реранкинг.

Tier 4 — Speech & Vision
ASR · TTS · OCR · VLM

WER ≤8% на корп. речи. OCR с layout (накладные, паспорта, договоры с печатями).

Tier 5 — Guardrails & Classifiers
PII · toxicity · prompt injection · routing

Защита от prompt injection, маскирование PII, классификатор для роутинга между копилотами.

Обязательные характеристики любой модели в стеке

  • ✓ Поддержка квантизации FP16 / INT8 / INT4
  • ✓ Fine-tuning на данных клиента (LoRA / QLoRA минимум)
